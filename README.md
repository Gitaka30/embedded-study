RTOSの動き

【 1ミリ秒（1000μs）の間に起きている CPU の動き 】

 時間(μs)  0        10                  400  410                 1000
   時間軸  +--------+--------------------+----+--------------------+---->
           |        |                    |    |                    |
 タスク    |        |                    |    |                    |
 =======   v        v                    v    v                    v
 [優先度:高]
 レーザー  [①発射]
 制御      |=======> (以降、1ms後までSleep/ブロック状態) ----------> [②発射]
           |
 [優先度:中]
 データ    |        [待ち伏せ]                 [④データ処理]
 受信      |        |===================> [ON]|====> (再びSleep)   |
           |        | (キューの空き待ち) |    |                   |
           |        |                    |    |                    |
 [優先度:低]        |                    |    |                    |
 Idle      |        |   [③アイドル待機] |   |                    |
 タスク    |        |===>=============>  |    |                    |
           |                             |    |                    |
 ──────────┼────────┼───────────────────┼──────┼────────────────┼──────────
 外部の    │        │                 │    │                  │
 ハード    │        │                 [光が戻る]                │
 動き      │        │                │[割り込み発生]           │

 AA図の解説（流れの補足）
0μs： 最優先の「レーザー制御」がCPUを奪い、レーザーをパッと【①発射】します。仕事が終わると、このタスクは1ms後まで休止（Sleep）に入ります。

10μs〜： 次に優先度の高い「データ受信」にCPUが移りますが、まだデータが届いていないので「キューが空だな」と判断し、データが来るまで【待ち伏せ】（ブロック状態）に入ります。

〜400μs： 動かすタスクがなくなったので、OSは自動的に【③アイドル待機】（Idleタスク）に切り替わり、CPUを休ませながら光が戻るのを待ちます。

400μs： 障害物に反射した光がセンサーに届き、ハードウェアからマイコンへ【割り込み】が入ります。ここで受信した点群データがキューへ投げ込まれます。

410μs： キューにデータが入った瞬間、待ち伏せしていた「データ受信タスク」がガバッと目を覚まし、CPUが切り替わって【④データ処理】（障害物の判定など）を猛スピードで実行します。処理が終わると、再び待ち伏せ状態に戻ります。

1000μs（1ms後）： きっちり1msが経過したため、タイマーによって「レーザー制御」が強制的に起こされ、【②発射】へとループします。

このように、RTOSは「割り込み」と「タスクの優先度」をきっちり管理することで、1ミリ秒という極小の時間の中でも、CPUを1コマも無駄にすることなく限界まで効率よく使い回しています。

この動きのイメージが頭に入っていれば、仕様書を見たときに「なぜここでキューが必要なのか」「なぜタスクの優先度をこう設定しているのか」が手に取るように分かるようになります。
